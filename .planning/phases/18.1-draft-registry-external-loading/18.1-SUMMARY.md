## What Shipped

### `crates/core/src/speculative/draft_registry.rs` (new, ~470 lines)

A runtime registry for heterogeneous external draft models. Decoupled from the
compile-time `ArchitectureRegistry` so drafts can be added/removed at runtime.

**Public API:**
- `DraftId(String)` — opaque user-supplied id (slug)
- `DraftSpec { id, model_dir, arch_hint, kv_blocks, ref_count }` — registration entry
- `LoadedDraft { spec, backend: Box<dyn ModelBackend>, block_allocator }` — loaded state
- `DraftState::{Unloaded, Loaded}` — state machine
- `DraftModelRegistry` — `RwLock<HashMap<DraftId, DraftState>>` with:
  - `register(spec)` — metadata-only, no I/O
  - `attach_loaded(id, backend)` — promotes `Unloaded → Loaded`, caller-driven
  - `unload(id)` — releases backend and block allocator
  - `lookup(id)` / `is_loaded(id)` / `contains(id)` — read-only state inspection
  - `increment_ref` / `decrement_ref` / `ref_count` — refcount primitives (used by 18.3)
  - `ids()` / `len()` / `is_empty()` — collection introspection
- `DraftRegistryError` — `UnknownDraftId`, `AlreadyLoaded`, `InUse`, `LoadFailed`

## Design Notes

1. **No vllm-model dependency in registry** — `DraftModelRegistry` does NOT
   depend on `vllm-model`. The actual `ModelLoader` call lives at the caller
   (server, test harness), which then hands a `Box<dyn ModelBackend>` to
   `attach_loaded`. This keeps the registry usable from contexts where
   `vllm-model` is unavailable (CUDA-less builds, custom backends).

2. **State machine over enum** — `DraftState::{Unloaded, Loaded}` makes the
   lazy semantics explicit. `register(spec)` always starts in `Unloaded`;
   `attach_loaded` is the only path to `Loaded`; `unload` returns to
   `Unloaded` and drops the backend (allocator is dropped → all blocks free).

3. **Block allocator isolation** — each `LoadedDraft` owns its own
   `BlockAllocator`, sized by `spec.kv_blocks`. No code path allows a draft's
   block id to be allocated from the target's `MemoryManager` or another
   draft's allocator. State leak is impossible by construction.

4. **Refcount primitives present but unused in 18.1** — `increment_ref` /
   `decrement_ref` are stored but auto-unload-on-zero is deferred to 18.2
   (LIFE-03). For now, `unload` is unconditional; refcounts are informational.

5. **`Engine::with_drafts` panics on duplicate ids** — duplicate ids in a
   single spec list is treated as a programmer error. Runtime `register_draft`
   returns `DraftRegistryError::AlreadyLoaded` so callers can react.

## Deferred to Phase 18.2+

- `unload` does not check refcount — auto-unload on zero lands in 18.2 (LIFE-03)
- Memory budget enforcement (MEM-01..03) lands in 18.2
- Request → draft routing (RTE-01..03) lands in 18.3
