# Phase 18.1: Draft Registry + External Loading - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning
**Mode:** Autonomous (smart-discuss)

<domain>
## Phase Boundary

Introduce a runtime `DraftModelRegistry` that holds zero-or-more external draft models alongside the target. Each draft is an independent `ModelBackend` instance loaded from its own checkpoint (potentially different architecture and/or size from the target). Weights are lazily loaded on first use. The registry exposes `register`, `load`, `unload`, and lookup-by-id at runtime, decoupled from the existing `ArchitectureRegistry` (which is a static, compile-time construct). v17 self-spec baseline remains the fallback path.

This phase delivers:
- A `DraftModelRegistry` data structure holding heterogeneous draft instances keyed by `DraftId`
- Lazy weight loading ‚Äî `register` is metadata-only; first request that names a draft triggers `load`
- Each draft owns its own `ModelBackend` plus a dedicated KV cache block pool (no state leak with target or other drafts)
- Integration with the existing `Engine` config (`EngineConfig`) ‚Äî drafts are declared as a `Vec<DraftSpec>` and made available via the registry
- Heterogeneous coexistence: target=Llama + draft=Qwen, target=Large + draft=Small, etc., with no conflict

</domain>

<decisions>
## Implementation Decisions

### DraftId / DraftSpec shape
- `DraftId` is a newtype `pub struct DraftId(pub String)` (UUID or user-supplied slug)
- `DraftSpec` is the registration entry: `{ id: DraftId, model_dir: PathBuf, arch_hint: Option<String>, kv_blocks: usize }`
- Each `DraftSpec` is fully self-describing ‚Äî registry holds `Vec<DraftSpec>` pre-load and `HashMap<DraftId, LoadedDraft>` post-load

### Registry storage and locking
- `DraftModelRegistry` is a thin wrapper around `RwLock<HashMap<DraftId, LoadedDraft>>` plus a `Vec<DraftSpec>` for pending registrations
- Reads (lookup) use read lock; writes (register/load/unload) use write lock
- No background workers in this phase ‚Äî explicit calls only

### Lazy loading trigger
- Lazy = `LoadedDraft` has an `Unloaded(DraftSpec)` and `Loaded { backend, kv_blocks }` variant
- Lookup method `get_or_load(&self, id: &DraftId) -> Result<&LoadedDraft>` triggers load on `Unloaded` and atomically transitions to `Loaded` via write lock
- Loading failure leaves the entry in `Unloaded` state and propagates `DraftLoadError` so the caller can fall back (FALL-01 wiring deferred to Phase 18.3)

### KV cache isolation
- Each loaded draft gets its own block allocator sub-pool sized by `kv_blocks` from `DraftSpec`
- The registry's `LoadedDraft` owns its block allocator ‚Äî no sharing with target's `MemoryManager` or other drafts
- Drafts register themselves with target's prefix-cache *only if* user explicitly opts in (NOT in this phase ‚Äî out of scope, deferred)

### Engine integration shape
- `Engine::new(config)` reads `config.drafts: Vec<DraftSpec>` and passes them to a fresh `DraftModelRegistry`
- `engine.draft_registry()` returns `&DraftModelRegistry` for runtime manipulation
- No new engine methods for routing ‚Äî Phase 18.3 wires `Request.draft_model_id` ‚Üí registry lookup. Phase 18.1 only delivers the registry + lazy-load primitives.

### Relationship to ArchitectureRegistry
- `DraftModelRegistry` is independent ‚Äî uses the same underlying `ModelLoader` machinery (and thus the same `ArchitectureRegistry`) but does not extend it
- Rationale: `ArchitectureRegistry` is static (compile-time registration), `DraftModelRegistry` is dynamic (runtime register/unload). Keeping them separate preserves the compile-time dispatch invariants.

### Error types
- New `DraftRegistryError` enum in `crates/core/src/speculative/draft_registry.rs` with variants `UnknownDraftId`, `AlreadyLoaded`, `LoadFailed(String)`, `InUse` (for unload-with-refcount>0 case in 18.2)
- Implements `std::error::Error` via `thiserror`

### the agent's Discretion
- Exact `LoadedDraft` field layout (whether to pre-allocate KV tensors vs lazy)
- Internal ordering of `register` calls (whether order is preserved, dedup behavior)
- Whether `DraftSpec` exposes a `name()` getter for log messages

</decisions>

## Existing Code Insights

### Reusable Assets
- `vllm_model::loader::ModelLoader` (and `ModelLoaderBuilder`) ‚Äî full checkpoint loading pipeline, supports safetensors + GGUF, returns `Box<dyn ModelBackend>`
- `vllm_model::arch::registry::ArchitectureRegistry` ‚Äî auto-detects architecture from `config.json` and dispatches to the correct `Architecture` impl
- `vllm_core::scheduler::memory::MemoryManager` ‚Äî block allocator with `allocate`, `free`, `rollback`. Pattern to mirror for per-draft block pools.
- `vllm_traits::ModelBackend` ‚Äî the trait drafts must implement (already target does)

### Established Patterns
- Builder pattern: `ModelLoaderBuilder`, `SpeculationConfigBuilder` ‚Äî follow for `DraftSpec` if it grows complex
- thiserror for error enums (`EngineError`, `SpeculativeError` etc.)
- `RwLock<HashMap<...>>` is the standard in-memory map pattern in this codebase (see `ArchitectureRegistry`)
- Newtype wrappers (`SeqId(u64)`, `TokenId(u64)`, `BlockId(usize)`) ‚Äî apply same pattern for `DraftId`

### Integration Points
- `crates/core/src/engine.rs` ‚Äî `Engine::new` and `EngineConfig` ‚Äî registry passed in via config
- `crates/core/src/speculative/mod.rs` ‚Äî add `pub mod draft_registry;` and re-export the new public types
- `crates/core/src/types.rs` ‚Äî `Request` struct (Phase 18.3 will add `draft_model_id`, but we don't add it now ‚Äî Phase 18.1 only delivers the registry)

</code_context>

<specifics>
## Specific Ideas

- DraftId is a String slug (e.g., `"qwen-small"`, `"llama-tiny"`) not a numeric ID ‚Äî matches user mental model
- Lazy loading is keyed on first `get_or_load` call ‚Äî explicit `preload()` method also exposed for users who want eager init
- No `DraftModelRegistry` instance survives past the `Engine` ‚Äî it's owned by `Engine` and dies with it (simplifies lifecycle in 18.2)
- Drafts are read-only after load ‚Äî no `update_weights` API in this phase

</specifics>

## Deferred Ideas

- **Prefix-cache sharing with target** ‚Äî out of scope; would need new cache key design
- **Cross-GPU draft placement (draft on GPU 0, target on GPU 1)** ‚Äî v19.0+ MULTI-06
- **Hot-swap of draft mid-request** ‚Äî v19.0+ MULTI-04
- **Draft fine-tuning hooks** ‚Äî out of scope, training belongs elsewhere
- **Refcount-based auto-unload** ‚Äî Phase 18.2 (LIFE-03)
- **Memory budget enforcement** ‚Äî Phase 18.2 (MEM-01..03)
- **Request ‚Üí draft routing** ‚Äî Phase 18.3 (RTE-01..03)
- **External draft load failure ‚Üí fallback to self-spec** ‚Äî Phase 18.3 (FALL-01)

</deferred>
