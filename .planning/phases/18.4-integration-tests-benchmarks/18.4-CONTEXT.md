<domain>
## Phase Boundary

End-to-end validation of the v18.0 multi-model speculative decoding pipeline
through integration tests and a benchmark harness. No new public API ŌĆö this
phase composes the building blocks shipped in 18.1/18.2/18.3 into:

1. **Lifecycle integration tests** ŌĆö register ŌåÆ lazy load ŌåÆ route ŌåÆ unload
   exercised in one test, with the real registry + budget + resolver

2. **Budget boundary tests** ŌĆö accept (just-fits), refuse (just-over),
   concurrent N drafts (within budget), concurrent N+1 drafts (over budget)

3. **Mixed-draft routing test** ŌĆö two requests with different
   `draft_model_id` values in the same batch, each resolved to its own
   backend, both producing tokens

4. **Fallback paths** ŌĆö explicit tests for load-time failure (unknown id,
   loader returns Err, budget exceeded) and runtime failure (draft forward
   errors mid-step)

5. **Benchmark** ŌĆö `benchmarks/multi_draft_speculative.rs` compares
   single-target-only throughput vs target+external-draft throughput,
   reports VRAM budget state via the metrics snapshot

The phase uses a `MultiDraftTestHarness` that wires the real `DraftResolver`
+ `DraftModelRegistry` + `MemoryBudget` + stub loaders/backends, so the
tests exercise production code paths end-to-end without requiring GPU or
actual model weights.

</domain>

<decisions>
## Implementation Decisions

### Test harness location
- New file `crates/core/tests/multi_draft_integration.rs` ŌĆö integration tests
  live in `tests/` because they exercise the public surface across multiple
  modules (registry + resolver + metrics)
- The harness helpers (loaders, backends, builders) are defined in the test
  file itself, not in `vllm-testing`, because they're phase-specific

### Stub backends
- New `MultiDraftBackend` test type with deterministic behavior:
  - Records every `forward()` call (id, input)
  - Returns configurable outputs (for verifier tests)
  - Supports a `fail_next_n: AtomicU32` knob for FALL-02 testing
- Implements `ModelBackend` trait fully

### Benchmark scope
- Single benchmark: `benchmarks/multi_draft_speculative.rs`
- Compares three configurations:
  1. No draft (baseline) ŌĆö pure target decode
  2. Self-spec draft (v17 baseline) ŌĆö speculative with self
  3. External draft (v18.0) ŌĆö speculative with registered external
- Reports throughput (tokens/sec) and p50 latency for each
- VRAM budget metrics snapshot is printed but not benchmarked (it's a
  snapshot, not a workload)

### Integration test plan
- `test_lifecycle_register_lazy_load_route_unload` ŌĆö happy path
- `test_budget_accept_just_fits` ŌĆö exactly at boundary
- `test_budget_refuse_just_over` ŌĆö 1 byte over
- `test_budget_concurrent_n_drafts_within_budget`
- `test_budget_concurrent_n_plus_one_over_budget`
- `test_mixed_draft_routing_two_different_drafts_one_batch`
- `test_fallback_unknown_id_returns_self_spec`
- `test_fallback_loader_error_returns_self_spec`
- `test_fallback_runtime_error_marks_sequence_degraded`
- `test_unload_with_nonzero_refcount_returns_inuse`
- `test_force_unload_overrides_refcount`
- `test_decrement_ref_auto_unloads_at_zero`
- `test_get_loaded_backend_returns_arc_for_loaded_drafts`
- `test_metrics_snapshot_reflects_all_counters`

### Benchmark design
- Reuse the harness to spin up an Engine with three configurations
- Use the existing `TestFixtures::increment_engine_with` for warmup
- Run 50 iterations of "process 16 tokens" workload
- Measure wall-clock time per iteration
- Output Criterion-style throughput comparison

### Cargo registration
- Add new bench to `crates/core/Cargo.toml` `[[bench]]` section
- No new dependencies ŌĆö uses existing `criterion`, `tokio`, `vllm-testing`

### Test execution time
- Keep total integration test runtime under 5 seconds (these run in CI)
- Benchmark runs only via `cargo bench`, not in normal `cargo test`

### the agent's Discretion
- Exact stub backend behavior (deterministic counter + optional failure)
- Whether to include a chaos test (random load failures) ŌĆö skip for now,
  scope creep risk
- Whether the benchmark measures warm cache or cold cache ŌĆö chose cold cache
  for honesty about first-request latency

</decisions>
