# Code Review: Phase 18.4 (Multi-Model Speculative Decoding — Integration & Benchmarks)

## Summary

Phase 18.4 is a validation-only phase that adds `crates/core/tests/multi_draft_integration.rs`
(14 integration tests), `crates/core/benches/multi_draft_speculative.rs` (a Criterion
benchmark comparing three draft-routing configurations), and a `[[bench]]` entry in
`crates/core/Cargo.toml`. The tests exercise `DraftModelRegistry`, `DraftResolver`,
`MemoryBudget`, and `EnhancedMetricsCollector` end-to-end via stub backends.

**Cross-checked against underlying types** in
`crates/core/src/speculative/{draft_registry,draft_resolver,memory_budget,model}.rs` and
`crates/core/src/metrics/collector.rs`. Every assertion in the test file corresponds to
real behavior of the production code; no assertion is false-positive or trivially passes.
The Cargo.toml `[[bench]]` registration matches the on-disk filename and follows the
existing pattern of the other three benches in the same crate.

**No security issues** — there is no user input, no path I/O (despite the literal
`"/nope"` paths used as `model_dir`), no `unsafe`, and no eval/exec patterns. All locks
are standard `lock().unwrap()` against local `Mutex`es that never escape the test
scope.

**No critical bugs found.** The findings below are quality/coverage concerns:
misleading test names that obscure what is actually being verified, several gaps in
the assertions (assertions that would fail to catch regressions in adjacent code
paths), and minor stylistic issues in the bench and test files.

## Warnings

### WR-01: Test name claims sequence-degradation behavior that is not exercised

**File:** `crates/core/tests/multi_draft_integration.rs:317-339`
**Issue:** `test_fallback_runtime_error_marks_sequence_degraded` is named after
the FALL-02 engine behavior (marking a sequence as `degraded_draft=true` after a
draft runtime error). The comment at line 331-332 acknowledges this with
"simulated here; in real engine, the step loop catches and sets the flag". The
test body only verifies that the stub backend returns `Err` on first call and
`Ok` on the second — it never inspects any sequence-level state. A reader scanning
the test name will assume the FALL-02 path is covered when it is not. The test
will silently continue to pass even if the engine's degraded-flag handling
regresses.
**Suggestion:** Either rename the test to `test_stub_backend_fail_next_n` (matching
what it actually tests) or add a follow-up integration test that drives the real
engine step loop with a failing draft and asserts that the sequence's
`degraded_draft` flag is set. If the engine surface for that flag is not yet
publicly observable, note this as a coverage gap in the plan rather than letting
the test name overstate coverage.

### WR-02: Misleading "concurrent" naming — tests are strictly sequential

**File:** `crates/core/tests/multi_draft_integration.rs:240-277`
**Issue:** `test_budget_concurrent_n_drafts_within_budget` and
`test_budget_concurrent_n_plus_one_over_budget` are named "concurrent" but neither
spawns any threads — both run resolutions sequentially in a single thread. The
tests verify budget-boundary correctness, which is a real concern, but the name
implies they validate thread safety of the `MemoryBudget` / `DraftModelRegistry`
locks. A reader looking for concurrency regression coverage will be misled.
**Suggestion:** Rename to `test_budget_accepts_n_drafts_within_budget` and
`test_budget_refuses_n_plus_one_drafts_over_budget`. If true concurrency is
desired, add a separate `std::thread`-based test that resolves multiple drafts
from N threads and asserts the final `reserved_drafts_bytes` is the cumulative
sum.

### WR-03: `test_budget_concurrent_n_plus_one_over_budget` does not verify state consistency after fallback

**File:** `crates/core/tests/multi_draft_integration.rs:258-277`
**Issue:** After resolving draft `b` falls back to `SelfSpec` due to budget
exhaustion, the test asserts the metric counters but does **not** assert that
draft `b` remained `Unloaded` in the registry (`!h.registry.is_loaded(&DraftId("b".into()))`).
A regression that left draft `b` in a partially-loaded state (e.g., budget
reserved but state machine inconsistent) would not be caught.
**Suggestion:** Add an assertion like
`assert!(!h.registry.is_loaded(&DraftId("b".into())), "fallback must leave b Unloaded");`
to lock in the post-fallback invariant.

### WR-04: `test_lifecycle_register_lazy_load_route_unload` does not verify the loader was invoked

**File:** `crates/core/tests/multi_draft_integration.rs:200-212`
**Issue:** The test name advertises "lazy_load", which by convention means the
loader is deferred until first resolve. The test inserts a backend into
`MapLoader`, calls `resolve`, and asserts the result is `External`. But it does
not observe the `MapLoader` (e.g., by checking that the entry was removed from
its HashMap, or by adding a `load_count` field to `MapLoader`). A regression in
`DraftResolver::resolve` that bypassed the loader path (e.g., by accidentally
promoting `Unloaded` → `Loaded` without a backend) would still pass.
**Suggestion:** Either (a) add a `pub load_count: AtomicU64` to `MapLoader` and
assert it incremented after `resolve`, or (b) assert the loader's map is now
empty for that id after `resolve`.

### WR-05: `test_unload_with_nonzero_refcount_returns_inuse` does not verify state is preserved

**File:** `crates/core/tests/multi_draft_integration.rs:341-352`
**Issue:** The test asserts that `unload` returns `InUse(1)` but does not
assert that the draft remains `Loaded` afterward. The equivalent in-registry
test at `crates/core/src/speculative/draft_registry.rs:758-771` does check
`assert!(registry.is_loaded(...))` after the failed unload, which is the
invariant that prevents a regression where the failed unload corrupts the state
machine. The integration test should preserve the same invariant.
**Suggestion:** Add `assert!(h.registry.is_loaded(&DraftId("a".into())));` after
the `unwrap_err()` line.

### WR-06: `test_budget_concurrent_n_drafts_within_budget` uses weak assertion

**File:** `crates/core/tests/multi_draft_integration.rs:251`
**Issue:** The assertion `assert!(h.budget.snapshot().reserved_drafts_bytes > 0);`
is too weak — it would pass even if only one of five drafts reserved anything.
A more useful assertion is the exact expected cumulative reservation, which would
catch a regression where `attach_loaded_budgeted` skipped the budget reservation
for some entries.
**Suggestion:** Replace with
`assert_eq!(h.budget.snapshot().reserved_drafts_bytes, 5 * (16 * 1024 * 1024));`
(assuming each draft is `kv_blocks=1`, `weight_size=0`).

### WR-07: Bench measurement semantics inconsistent with group name

**File:** `crates/core/benches/multi_draft_speculative.rs:172-188`
**Issue:** The benchmark group is named `multi_draft_throughput`, but
`b.iter` returns `start.elapsed()` — i.e., wall time per 16-step iteration.
Criterion's throughput metric (iters/sec) is meaningful when each iteration is a
fixed unit of work, which is the case here. The two are not strictly
inconsistent, but the measurement name "throughput" suggests tokens-per-second
or steps-per-second, not "total wall time per 16-step batch". A reader
interpreting the bench output as steps/sec may be confused.
**Suggestion:** Either (a) compute and return per-step time
(`start.elapsed() / 16`) and rename the group `multi_draft_step_latency`, or
(b) keep the wall-time measurement and rename the group `multi_draft_iter_total`
or similar. Pick a name that matches the measurement.

## Info

### IN-01: Unnecessary `#[allow(clippy::needless_range_loop)]` at crate level

**File:** `crates/core/tests/multi_draft_integration.rs:7`
**Issue:** The file has `#![allow(clippy::needless_range_loop)]` but never uses
any `for i in 0..n` range loop with a non-loop-counter index. All loops in the
file iterate over named slices (e.g., `for name in ["a", "b", ...]`) or simple
counting loops (`for _ in 0..16` in the bench, not the tests). The allow is
dead and adds noise to `cargo clippy --all-targets` output.
**Suggestion:** Delete the line. If it was inherited from a copy-pasted
template, confirm via `rg "for .* in 0\\.\\." crates/core/tests/multi_draft_integration.rs`
that no range loop actually needs it.

### IN-02: `BenchLoader::Default` impl is never used

**File:** `crates/core/benches/multi_draft_speculative.rs:114-118`
**Issue:** `impl Default for BenchLoader` is defined but `BenchLoader` is only
constructed via `BenchLoader::new()` inside `make_harness`. The `Default` impl
adds API surface without benefit.
**Suggestion:** Delete the `Default` impl.

### IN-03: Explicit pre-warmup duplicates Criterion's internal warmup

**File:** `crates/core/benches/multi_draft_speculative.rs:178-180`
**Issue:** The bench calls `run_iteration` 5 times before `group.bench_with_input`,
but Criterion already runs its own warmup phase (default 3 seconds) before
measurement. The explicit warmup is harmless but redundant for this benchmark
since the work per iteration is tiny (a few atomic ops + one HashMap lookup).
**Suggestion:** Remove the explicit warmup loop, or document why the manual
warmup is needed (e.g., to amortize first-load cost for `external_draft`, which
is a legitimate concern but is not currently called out).

### IN-04: `MapLoader::fail_on` is checked but never populated in any test

**File:** `crates/core/tests/multi_draft_integration.rs:113-153`
**Issue:** `MapLoader` exposes `will_fail(id)` and a `fail_on: Mutex<Vec<DraftId>>`
field, but no test calls `will_fail`. The "loader fails on load" path is
exercised instead by simply not inserting the backend (test
`test_fallback_loader_error_returns_self_spec` at line 308-315). The
`will_fail` machinery is dead code that adds surface area and a second lock
acquisition per `load` call.
**Suggestion:** Either delete `fail_on` and `will_fail` from `MapLoader`, or
add a test that uses `will_fail` to differentiate between "loader has no entry"
and "loader returned an error" — the two are semantically distinct and may want
separate metrics handling.
