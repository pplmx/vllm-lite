### `crates/core/benches/multi_draft_speculative.rs` (new, ~190 lines)

Criterion benchmark comparing three configurations:
- `no_draft` ‚Äî pure target decode via resolver fallback path
- `self_spec` ‚Äî v17 baseline (self-spec layer)
- `external_draft` ‚Äî v18.0 new path (registered external model)

**Benchmark methodology:**
- 5 warmup iterations + measured iterations
- 16 decode steps per iteration (simulates realistic decode batch)
- Each step: resolve draft via `DraftResolver` ‚Üí forward through resolved
  backend
- `BenchBackend` produces tokens that mix in the backend id (so different
  backends produce distinguishable outputs)

**Sample results** (release profile, --quick):
```
multi_draft_throughput/no_draft       ~1.67 ¬µs / 16 steps
multi_draft_throughput/self_spec      ~1.70 ¬µs / 16 steps
multi_draft_throughput/external_draft ~2.13 ¬µs / 16 steps
```

The v18.0 external-draft path adds ~25% orchestration overhead per 16-step
iteration vs the no-draft baseline. This is consistent with the extra work
of: registry lookup, loader invocation, `attach_loaded_budgeted` path. In
production this overhead is amortized over real forward passes (each step
does ¬µs-to-ms of inference), so the orchestration cost is in the noise.

## Design Notes

1. **Stub backends are stateful, not just typed** ‚Äî `StubBackend` tracks call
   count and supports per-instance `fail_next_n`. This makes the harness
   expressive enough to test runtime error paths without invasive mocking.

2. **Loader is a real `DraftLoader` impl, not a mock** ‚Äî `MapLoader` is a real
   implementation of the trait, so the integration tests exercise the same
   code path the production server uses.

3. **Benchmark measures orchestration, not inference** ‚Äî explicitly called out
   in the module docs. The 25% overhead on external_draft is the cost of
   registry lookup + loader invocation + Arc cloning, not actual model
   forward passes. Real inference dwarfs this.

4. **No new dependencies** ‚Äî reuses `criterion`, `tokio`, `vllm-testing`,
   `vllm-traits`, `vllm-core`. The bench uses only `std` collections +
   criterion + vllm_core APIs.

5. **`BenchBackend.id` is used, not silenced** ‚Äî mixed into token output so
   different backends produce distinguishable tokens. This makes the bench
   closer to real behavior (different drafts do produce different token
   streams) without adding complexity.
