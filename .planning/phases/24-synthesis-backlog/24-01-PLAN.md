**DISPATCH SYNTHESIS SUBAGENT:**

Pass it the 4 input SUMMARY.md files (lighter than full REPORTS) and ask it to:

1. **Cross-dimensional synthesis (SYNTH-01):**
   - Read all 4 SUMMARY.md
   - Identify root-cause patterns that span dimensions
   - Example: "Naming inconsistency in `kv_cache_fp8.rs` (orphan module)" may correlate with "no module doc" (DOCS-02) — both stem from "module not wired into module tree"
   - Group findings by root cause, not by audit dimension
   - Produce SYNTHESIS.md with:
     - Top root causes (5-10 most impactful)
     - Cross-references: which audit findings map to which root cause
     - Hot spots: modules/files with multiple findings

2. **Prioritized backlog (SYNTH-02):**
   - Read all 4 SUMMARY.md P0/P1/P2 tables
   - Consolidate into single BACKLOG.md
   - Columns: ID | Description | Source dimension(s) | Severity | Impact | Effort (hours) | Suggested phase (v20.x)
   - Sort by severity (P0 first), then by impact
   - Top of file: count summary
   - Group by suggested phase

3. **v20.0+ migration roadmap (SYNTH-03):**
   - Group BACKLOG items into 3-6 proposed v20.x phases
   - Each phase should:
     - Have a clear goal
     - Be self-contained enough to ship independently
     - Build on prior phase (dependency chain)
     - Target ~1-2 weeks of work
   - Suggested phase shape (advisory):
     - Phase 20.1: P0 fixes only (critical security/correctness)
     - Phase 20.2: Architectural restoration (layering, God modules)
     - Phase 20.3: Error handling standardization
     - Phase 20.4: Doc coverage + external doc fix
     - Phase 20.5: Naming cleanup + module consolidation
     - Phase 20.6: Polish (deprecation, comments)
   - Adjust based on actual findings
</action>
