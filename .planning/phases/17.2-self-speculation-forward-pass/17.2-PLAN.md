# Phase 17.2 Plan: Self-Speculation Forward Pass

**Created:** 2026-05-13
**Status:** Planned
**Requirements:** SELF-01, SELF-02, SELF-03
**Success Criteria:** 3/3

## Summary

Replace the stub `SelfSpeculativeModel.generate_draft()` with a real layer-truncated forward pass. Add `forward_to_layer()` to the `ModelBackend` trait and implement it across all concrete models (Qwen3, Llama, Mistral, etc.). Create an isolated draft KV cache in `SelfSpeculativeModel` with its own block pool, separate from the scheduler-managed target blocks. Use greedy argmax sampling for deterministic draft generation. Weight sharing uses zero-copy `Arc<Tensor>` references (clone is O(1) since candle Tensors are Arc-based).

## Plans

### Plan 17.2-A: Add `forward_to_layer()` to `ModelBackend` Trait

**Files:** `crates/traits/src/model.rs`, `crates/traits/src/lib.rs`
**Requires:** SELF-01
**Depends on:** None

Add a new method `forward_to_layer()` to the `ModelBackend` trait with a default implementation that falls back to the full `forward()`. This allows concrete models to override with a layer-truncated forward.

- Add trait method signature:

  ```rust
  fn forward_to_layer(
      &mut self,
      seq_ids: &[SeqId],
      input_tokens: &[Vec<TokenId>],
      positions: &[Vec<usize>],
      kv_block_ids: &[Vec<usize>],
      num_computed_tokens: &[usize],
      is_prefill: &[bool],
      upto_layer: usize,
  ) -> Result<BatchOutput>;
  ```

- Default impl: ignore `upto_layer`, call `self.forward()` (safe for draft models that don't support truncation)
- Update `BoxedModelBackend` wrapper in `crates/core/src/engine.rs` to delegate `forward_to_layer()` to inner model

### Plan 17.2-B: Implement `forward_to_layer()` for All Concrete Models

**Files:** `crates/model/src/qwen3/model.rs`, `crates/model/src/llama/model.rs`, `crates/model/src/mistral/model.rs`, etc.
**Requires:** SELF-01, SELF-02
**Depends on:** Plan 17.2-A

Implement layer-truncated forward in each concrete model. The pattern: iterate `self.layers.iter_mut().take(upto_layer)`, then apply final norm + lm_head to produce logits, and sample via argmax.

- **Pattern for each model's `forward_to_layer()`:**
    - Same embedding logic as `forward()` (or `forward_with_cache()`)
    - Layer loop: `for layer in self.layers.iter_mut().take(upto_layer) { ... }`
    - Apply `self.norm.forward(&hidden)` on truncated hidden states
    - Apply `self.lm_head.forward()` for logits
    - Greedy argmax sampling on last position's logits -> `BatchOutput`
    - Return `BatchOutput { seq_ids, next_tokens }`

- **Models to implement:**
    - `Qwen3Model` (`crates/model/src/qwen3/model.rs`) — uses forward_prefill/forward_decode per layer
    - `LlamaModel` (`crates/model/src/llama/`) — similar layer iteration with RMSNorm
    - `MistralModel` (`crates/model/src/mistral/`) — similar with sliding window attention
    - `MixtralModel` (`crates/model/src/mixtral/`) — MoE layers
    - `Qwen3_5Model`, `Gemma4Model` — their custom forward patterns
    - `Phi4Model` (`crates/model/src/phi4/`)
    - `Llama4Model`, `Gemma3Model`, `MistralSmallModel` — all need the trait impl

- **Key insight:** Each model's layer loop uses model-specific methods (e.g., `forward_prefill` vs `forward_decode` on Qwen3 layer blocks). The implementation needs to handle both prefill and decode phases within the truncated layer count. For decode, only the last token position matters.

- **Fallback for model-specific blocks:** The `TransformerBlock` trait's `forward()` is generic enough. For models whose blocks don't implement `TransformerBlock`, use the model's `forward_with_cache()` prefill/decode pattern but capped at `upto_layer`.

### Plan 17.2-C: Draft KV Cache Isolation in SelfSpeculativeModel

**Files:** `crates/core/src/speculative/self_spec.rs`
**Requires:** SELF-03
**Depends on:** Plan 17.2-A, Plan 17.2-B

Create an isolated `PagedKvCache` inside `SelfSpeculativeModel` with `draft_layer_count` layers Believing paralleling configurations as target. The draft KV cache has its own block pool, managed independently from the scheduler.

- Add fields to `SelfSpeculativeModel<M>`:

  ```rust
  draft_kv_cache: Option<PagedKvCache>,
  draft_kv_block_ids: HashMap<SeqId, Vec<usize>>,
  ```

- Initialize `draft_kv_cache` in `SelfSpeculativeModel::new()`:
    - Get draft layer count from `config.draft_layers` or default to `max(1, total_layers / 8)`
    - Create `PagedKvCache::new(draft_layer_count, model.num_heads(), head_dim, num_draft_blocks, device, quantized)`
    - Number of draft blocks can be smaller than target (e.g., 1/8 of target blocks)
- Track per-sequence block IDs in `draft_kv_block_ids: HashMap<SeqId, Vec<usize>>`
- Update `generate_draft()` to use `draft_kv_cache`:
    - Pass `draft_kv_block_ids` instead of target KV block IDs
    - After draft generation, store any newly allocated block IDs

**Challenge:** `PagedKvCache` lives in `crates/model` and requires candle. `SelfSpeculativeModel` lives in `crates/core` and doesn't depend on candle. Solution options:

1. **Move draft KV cache to the model side**: Have the concrete model implementation manage its draft KV cache internally. The `SelfSpeculativeModel` stores blocks count config and passes them through.
2. **Add candle feature gating**: Conditionally compile the draft cache field with `#[cfg(feature = "candle")]`.
3. **Store as opaque `Box<dyn Any>`**: Avoid candle dependency in core.
4. **Store block config + delegate to model**: Pass `(num_layers, num_heads, head_dim, num_blocks)` to the model and let it allocate.

**Recommended approach (option 4):** `SelfSpeculativeModel` holds `draft_kv_config: Option<(usize, usize, usize, usize)>` (layers, heads, head_dim, num_blocks) and the model allocates its own draft KV cache internally when `forward_to_layer()` is called with a new seq_id for the first time. This avoids dragging candle into `crates/core`.

Actual approach for this plan: **Add `PagedKvCache` as a feature-gated external dependency in `vllm-core`** OR **move `PagedKvCache` management to the model side via a new trait method**. Since `crates/core` already has `#[cfg(feature = "cuda-graph")]` feature gates referencing `vllm_model`, we can add a similar pattern. However, the cleanest design is:

- Add `forward_draft()` to `ModelBackend` trait in `crates/traits` that accepts draft-specific parameters
- The concrete model (with candle) manages its own draft KV cache internally
- `SelfSpeculativeModel` stores just the config and delegates to the model

**Implementation approach for this plan:** Keep it simple — make `SelfSpeculativeModel` manage `PagedKvCache` by having `crates/core` depend on `crates/model` under a feature flag, or alternatively, store the draft KV cache config and pass `draft_layer_count` to `forward_to_layer()` so the model can allocate its own per-sequence draft KV state.

Given the project's existing patterns (`draft_model: Option<Arc<Mutex<dyn ModelBackend>>>`), the simplest approach is:

- Add `fn supports_draft_cache(&self) -> bool` to `ModelBackend`
- Add `fn init_draft_kv_cache(&mut self, num_blocks: usize)` to `ModelBackend`
- Concrete models implement these, allocating the `PagedKvCache` internally

### Plan 17.2-D: Real `generate_draft()` Implementation

**Files:** `crates/core/src/speculative/self_spec.rs`
**Requires:** SELF-01, SELF-02, SELF-03
**Depends on:** Plan 17.2-A, Plan 17.2-B, Plan 17.2-C

Replace the stub `generate_draft()` that returns empty drafts with real draft generation using the truncated forward pass.

- `generate_draft()` implementation:

  ```text
  for each seq in batch:
      for pos in 0..num_tokens:
          input = current_tokens[seq]
          output = model.forward_to_layer(seq, input, ..., upto_layer=draft_layer_count)
          token = argmax(output.logits)    // greedy
          draft_tokens[seq].push(token)
          current_tokens[seq].push(token)
  ```

- Use the existing `sample_token()` method for argmax (temperature=0 branch)
- Track per-sequence draft KV block IDs through successive positions
- Wire in the isolated draft KV cache: before each `forward_to_layer()` call, pass the draft block IDs
- After generating all drafts, return `Vec<(SeqId, Vec<TokenId>)>`
- The `generate_draft()` needs access to the model's `forward_to_layer()` — currently it only has `&self`, but `ModelBackend::forward_to_layer()` takes `&mut self`. Change signature to `&mut self` or use interior mutability.
- **Decision:** Change `generate_draft()` to `&mut self` — this is consistent with how the engine calls the draft model via `draft_model.lock().unwrap().forward()`.

### Plan 17.2-E: Weight Sharing (Zero-Copy)

**Files:** `crates/core/src/speculative/self_spec.rs`
**Requires:** SELF-02
**Depends on:** Plan 17.2-A

The draft model doesn't allocate new weight tensors — it references the target model's layers via `Arc<Tensor>` cloned references. Since candle's `Tensor` is `Arc<TensorInner>` internally, `clone()` is O(1).

- In `SelfSpeculativeModel::new()`:
    - Don't clone the model at all — use the target model's layers directly via the `forward_to_layer(upto_layer=N)` mechanism
    - `SelfSpeculativeModel` wraps `model: M` and limits the forward pass by passing `upto_layer= draft_layer_count` to the model's `forward_to_layer()`
    - No additional GPU memory is consumed for weights
- Weight sharing is **implicit**: since we pass `upto_layer` to the same model instance, we're naturally using the same weights. No explicit sharing setup needed.
- The `model()` and `mut_model()` accessors expose the wrapped model for when full forward is needed (e.g., during verification or target forward)

**Eliminated the need for zero-copy cloning:** The design uses a single model instance with layer truncation. The `SelfSpeculativeModel` wraps the target model and limits forward depth. The draft and target are the same model — they inherently share all weights.

## Risk Assessment

| Risk                                                                        | Impact                    | Likelihood | Mitigation                                                                          |
| --------------------------------------------------------------------------- | ------------------------- | ---------- | ----------------------------------------------------------------------------------- |
| `forward_to_layer()` trait change breaks compilation across 10+ model impls | Build failure             | High       | Default impl falls back to `forward()` — models compile without change, opt-in only |
| Draft KV cache allocation in `crates/core` requires candle                  | Crate dependency issue    | Medium     | Delegate draft cache to model side via trait methods; use `#[cfg(feature)]`         |
| `generate_draft()` needs `&mut self` but `DraftVerifier` uses `&self`       | Compile error             | High       | Change trait signature to `&mut self` — affects 3 impls (self_spec, mock, model)    |
| Draft layer count of 1/8 for small models (< 8 layers) rounds to 0          | Silent bug                | Low        | Use `max(1, total_layers / 8)` — already in context                                 |
| Multiple model-specific block forward methods (prefill vs decode)           | Implementation complexity | Medium     | Only support decode phase for draft; prefill already handled by full target model   |
