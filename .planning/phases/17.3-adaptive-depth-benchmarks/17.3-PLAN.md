## Summary

## Plans

### Plan 17.3-A: EWMA Smoothing in DraftAccuracyTracker

**Files:** `crates/core/src/speculative/adaptive.rs`, `crates/core/src/types.rs`
**Requires:** ADPT-01
**Depends on:** None

Replace (or augment) the sliding-window `acceptance_rate()` in `DraftAccuracyTracker` with an EWMA-smoothed rate. The sliding window remains available for debugging but the decoder's `maybe_adjust()` uses the EWMA value.

- Add fields to `DraftAccuracyTracker`:
  - `ewma_alpha: f32` — smoothing factor (default 0.1)
  - `smoothed_rate: Option<f32>` — EWMA state, `None` until first observation
- Add `AdaptiveDraftConfig::ewma_alpha` field with default `0.1`
- On `record()`: update `smoothed_rate` via `ewma = alpha * current_rate + (1 - alpha) * previous_ewma`
  - On first observation, initialize `smoothed_rate = current_rate` (avoid initialization bias)
- Add `fn acceptance_rate_ewma(&self) -> f32` public method
- In `maybe_adjust()`: replace `self.accuracy_tracker.acceptance_rate()` with `self.accuracy_tracker.acceptance_rate_ewma()`
- Keep the original `acceptance_rate()` (sliding window) for backward compat and debugging
- Update tests to verify EWMA behavior:
  - `test_ewma_initialization` — first rate sets smoothed_rate directly
  - `test_ewma_convergence` — repeated same rate converges to that rate
  - `test_ewma_smoothing` — oscillating rates produce smoother output than sliding window

### Plan 17.3-B: Deadband Hysteresis in maybe_adjust()

**Files:** `crates/core/src/speculative/adaptive.rs`, `crates/core/src/types.rs`
**Requires:** ADPT-02
**Depends on:** Plan 17.3-A

Add deadband hysteresis to `AdaptiveSpeculativeDecoder::maybe_adjust()` so the decoder only adjusts when the EWMA-smoothed rate deviates beyond `target ± threshold`. Reset the cooldown counter on no-change decisions.

- Add `deadband_threshold: f32` to `AdaptiveDraftConfig` (default 0.05)
- Modify `maybe_adjust()`:
  - Read `rate = self.accuracy_tracker.acceptance_rate_ewma()`
  - Compute `deviation = rate - target`
  - If `deviation.abs() <= threshold`: no adjustment, **reset** `steps_since_adjustment = 0` (prevents stale cooldown)
  - If `deviation > threshold` (rate too high): increase draft count by `adjustment_step`, reset cooldown
  - If `deviation < -threshold` (rate too low): decrease draft count by `adjustment_step`, reset cooldown
- Replace the old hardcoded `0.1` thresholds with the configurable `deadband_threshold`
- Update existing tests for the new threshold behavior
- Add tests:
  - `test_deadband_no_adjustment` — steady rate within deadband → no change
  - `test_deadband_resets_cooldown` — within deadband resets `steps_since_adjustment` to 0
  - `test_deadband_above_triggers_increase` — rate > target + threshold → increase

### Plan 17.3-D: Benchmarks Crate — Speculative Benchmark Binary

**Files:** `benches/speculative.rs` (new), `benches/Cargo.toml`
**Requires:** BENCH-01, BENCH-02, BENCH-03, BENCH-04
**Depends on:** Plans 17.3-A, 17.3-B, 17.3-C

Add a new criterion benchmark file `benches/speculative.rs` registered in `benches/Cargo.toml` that measures speculative vs non-speculative decoding performance. Use `FakeModel` (already available in `vllm_core::engine::speculative::tests`) to construct an engine and measure throughput under different workloads.

#### Benchmark Structure

```rust
criterion_group!(speculative_benches, /* all bench fns */);
criterion_main!(speculative_benches);
```

#### Benchmark: `spec_non_speculative_comparison`

- **Config:** A/B comparison across 3 modes:
  1. Non-speculative (`step_with_draft(None)`)
  2. Fixed-depth speculative (`step_with_draft(Some(5))`) 
  3. Adaptive speculative (`step_adaptive_speculative()` with `AdaptiveSpeculativeDecoder`)
- **Model:** Two `FakeModel` instances — target and draft — where draft has configurable accuracy (e.g., 80% match rate by having draft produce the same token as target 80% of the time)
- **Workload:** Single sequence, 128 token output
- **Metrics:** P50/P95/P99 latency, tokens/sec throughput

#### Benchmark: `spec_multi_sequence_4`

- Like above but with 4 concurrent sequences in the engine
- Measures how speculative decoding behaves under batching

#### Benchmark: `spec_multi_sequence_16`

- Like above but with 16 concurrent sequences
- Tests scheduler + speculative interaction under load

#### Benchmark: `spec_warmup_vs_cold`

- Run speculative benchmark with and without warmup iterations
- Warmup: 10 iterations before measurement (criterion convention)
- Cold: skip warmup, start measuring immediately
- Demonstrates warmup impact on P50/P95

#### Output Format

Each benchmark prints an A/B comparison table:

```
| Mode               | P50 (ms) | P95 (ms) | P99 (ms) | Tok/sec | Spec Eff | Accept Rate |
|--------------------|----------|----------|----------|---------|----------|-------------|
| Non-speculative    | 12.3     | 15.1     | 18.2     | 81.2    | 1.00x    | —           |
| Speculative (fixed)| 10.1     | 12.4     | 14.8     | 99.0    | 1.22x    | —           |
| Adaptive           | 9.8      | 12.0     | 14.3     | 102.0   | 1.26x    | 82%         |
```

Speculative efficiency = non-spec throughput / spec throughput.

#### Model Setup for Benchmarks

Create a helper function `make_fake_models(acceptance_rate: f32) -> (FakeModel, FakeModel)` that produces target and draft models where the draft matches the target with the given probability.

- Target model always returns token 42
- Draft model returns 42 with probability `acceptance_rate`, and a different token (99) otherwise
- This gives deterministic, measurable acceptance rates

#### Bench Cargo.toml Changes

Add `[[bench]]` entry:
```toml
[[bench]]
name = "speculative"
path = "speculative.rs"
harness = false
```

(No new dependencies needed — `vllm-core`, `vllm-traits`, and `criterion` are already available.)

## Risk Assessment

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| EWMA + deadband interactions cause oscillation at boundary conditions | Wrong draft depth | Low | Test edge cases: rate exactly at boundary, rapid oscillations |
| Benchmarks use FakeModel so results don't represent real GPU models | Misleading perf expectations | Medium | Document that these are microbenchmarks; real model benchmarks deferred |
| EWMA initialization bias if first few observations are anomalous | Slow initial adaptation | Low | Initialize EWMA with first observed rate, not 0 |
| Cooldown reset on deadband no-change may cause too-frequent checks | Churn | Low | Keep default cooldown_steps=5 as the primary oscillation guard |
