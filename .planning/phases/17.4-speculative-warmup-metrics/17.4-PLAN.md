**Created:** 2026-05-13
**Status:** Planned

## Summary

## Plans

### Plan 17.4-A: Draft Warmup After Target Prefill

**Files:** `crates/core/src/engine/speculative.rs`, `crates/core/src/engine.rs`
**Requires:** WARM-01, WARM-02
**Depends on:** None (phases 17.1-17.3 complete)

After the target model completes a prefill step, run a truncated draft model forward to populate the draft's KV cache for any sequences in speculative mode)Skip non-speculative sequences.

Integration point: `step_speculative_inner()` in `crates/core/src/engine/speculative.rs` — detect prefill via `batch.phase`, then run `draft_model.lock().unwrap().forward()` with the same input to populate draft KV cache.

- Add a warmup method to `Engine<M>`:
  ```rust
  fn warmup_draft_kv(&mut self, batch: &Batch) -> Result<()> {
      if !self.speculative_mode {
          return Ok(());
      }
      let draft_model = match &self.draft_model {
          Some(dm) => dm.clone(),
          None => return Ok(()),
      };
      // Only warmup for sequences that will enter speculative decode
      for (i, seq_id) in batch.seq_ids.iter().enumerate() {
          // Run draft model forward on the same input to populate its KV cache
          let _ = draft_model.lock().unwrap().forward(
              &[*seq_id],
              std::slice::from_ref(&batch.input_tokens[i]),
              std::slice::from_ref(&batch.positions[i]),
              std::slice::from_ref(&batch.kv_block_ids[i]),
              std::slice::from_ref(&batch.num_computed_tokens[i]),
              std::slice::from_ref(&batch.is_prefill[i]),
          )?;
      }
      Ok(())
  }
  ```
- Call `self.warmup_draft_kv(&batch)` in `step_speculative_inner()` right after `batch` is built, but only when `batch.phase == BatchPhase::Prefill`
- The draft model's KV cache blocks are populated via the same `forward()` call — the model implementation internally uses the provided `kv_block_ids` to write KV cache entries
- Use tracing::debug! for warmup logging:
  ```rust
  tracing::debug!(
      seq_count = batch.seq_ids.len(),
      "Draft KV cache warmed up after prefill"
  );
  ```

**Key assumption:** The draft model shares KV block IDs with the target run — since both read from the same seq's block IDs, the draft's warmup forward populates the KV entries for those blocks. The target and draft KV caches are separate (managed by PagedKvCache per-model), so they write to independent storage even though they use the same logical block IDs.

### Plan 17.4-B: Speculative Efficiency Metric

**Files:** `crates/core/src/metrics/enhanced.rs`, `crates/core/src/metrics/collector.rs`, `crates/core/src/metrics/exporter.rs`
**Requires:** MTRC-02
**Depends on:** None (independent of warmup)

Add `speculative_efficiency` (draft_tokens / total_tokens) to `EnhancedMetrics` and `EnhancedMetricsCollector`, and export it via Prometheus.

- Add atomics to `EnhancedMetrics`:
  ```rust
  pub speculative_efficiency_percent: Arc<AtomicU64>,
  ```
- Add getter:
  ```rust
  pub fn speculative_efficiency(&self) -> f64 {
      let draft = self.speculative_draft_tokens_generated.load(Ordering::Relaxed);
      let total = draft + self.speculative_draft_tokens_accepted.load(Ordering::Relaxed);
      if total == 0 { 0.0 } else { draft as f64 / total as f64 }
  }
  ```
  Wait — efficiency is `draft_tokens / total_tokens`, but `total_tokens = target_tokens + draft_tokens`. The `speculative_draft_tokens_generated` is draft, and `speculative_draft_tokens_accepted` is target. So:
  - `total_tokens = draft_generated + accepted_target`
  - `efficiency = draft_generated / total_tokens`
  - This gives what fraction of total tokens came from the draft (higher is better)

- Add to `EnhancedMetricsCollector`:
  ```rust
  speculative_efficiency: AtomicU64,
  ```
  with `record_speculative_efficiency()` method (stored as fixed-point per existing convention): `(efficiency * 100000.0) as u64`

- Add getter in collector: `"speculative_efficiency"` → returns the gauge value

- Add to `PrometheusExporter::export_to_string()`:
  ```prometheus
  # HELP speculative_efficiency Draft token efficiency ratio (0-1)
  # TYPE speculative_efficiency gauge
  speculative_efficiency {:.3}
  ```

- Wire `record_speculative_efficiency()` call into `step_speculative_inner()` at the end (after verification results are known), computing:
  ```rust
  let total_draft: usize = draft_outputs.iter().map(|d| d.len()).sum();
  let total_accepted: usize = accepted_counts.iter().sum();
  let total_tokens = total_draft + total_accepted;
  if total_tokens > 0 {
      let efficiency = total_draft as f64 / total_tokens as f64;
      self.enhanced_metrics.record_speculative_efficiency(efficiency);
  }
  ```

Note: `step_speculative_inner()` in `speculative.rs` currently uses `self.metrics: MetricsCollector`, not `self.scheduler.enhanced_metrics`. We need access to `EnhancedMetricsCollector` from the engine. Either:
- Add a convenience method on the engine that delegates to `self.scheduler.enhanced_metrics` (recommended)
- Or pass the enhanced metrics collector through

**Approach:** Add a method `record_spec_efficiency(&self, efficiency: f64)` to the engine that calls `self.scheduler.enhanced_metrics.record_speculative_efficiency(efficiency)`.

### Plan 17.4-D: Per-Request Acceptance Rate Tracking

**Files:** `crates/core/src/engine/speculative.rs`, `crates/core/src/metrics/collector.rs`
**Requires:** MTRC-01
**Depends on:** None

Aggregate acceptance rate per-request using the existing `record_speculative_draft()` / `speculative_acceptance_rate()` counters in `EnhancedMetricsCollector`.

The existing infrastructure already handles aggregate tracking:
- `EnhancedMetricsCollector::record_speculative_acceptance(accepted, total)` — stores the current aggregated rate as a gauge
- `EnhancedMetricsCollector::speculative_acceptance_rate` gauge — already exported via Prometheus

What's missing: per-request tracking (not just aggregate). Add:

- Add a field to track per-request acceptance:
  ```rust
  // In EnhancedMetricsCollector
  per_request_acceptance: DashMap<SeqId, (AtomicU64, AtomicU64)>, // (accepted, total)
  ```

- Add methods:
  ```rust
  pub fn record_per_request_acceptance(&self, seq_id: SeqId, accepted: usize, total: usize) {
      let entry = self.per_request_acceptance
          .entry(seq_id)
          .or_insert_with(|| (AtomicU64::new(0), AtomicU64::new(0)));
      entry.0.fetch_add(accepted as u64, Ordering::Relaxed);
      entry.1.fetch_add(total as u64, Ordering::Relaxed);
  }

  pub fn get_per_request_acceptance_rate(&self, seq_id: SeqId) -> f64 {
      self.per_request_acceptance
          .get(&seq_id)
          .map(|entry| {
              let accepted = entry.0.load(Ordering::Relaxed);
              let total = entry.1.load(Ordering::Relaxed);
              if total == 0 { 0.0 } else { accepted as f64 / total as f64 }
          })
          .unwrap_or(0.0)
  }

  pub fn remove_per_request(&self, seq_id: SeqId) {
      self.per_request_acceptance.remove(&seq_id);
  }
  ```

- Wire into `step_speculative_inner()`:
  - After verification, for each sequence, call `self.record_per_request_acceptance(seq_id, accepted, total_drafts_for_seq)`
  - When sequences finish (in the finished-sequences cleanup block), call `self.remove_per_request(seq_id)`

- Expose via Prometheus with per-seq_id labels (optional — Prometheus high cardinality warning applies). For now, export as a summary gauge `speculative_acceptance_rate` (unchanged — already aggregated) plus add a `speculative_per_request_count` gauge for the number of tracked requests.

### Plan 17.4-E: Warmup Integration in Engine Step

**Files:** `crates/core/src/engine/speculative.rs`
**Requires:** WARM-01, WARM-02
**Depends on:** Plan 17.4-A

Wire the draft warmup into the real engine step flow with proper phase detection and conditional execution.

- In `step_speculative_inner()`:
  ```rust
  fn step_speculative_inner(&mut self, max_draft: usize) -> Result<Vec<(SeqId, TokenId)>> {
      let start = std::time::Instant::now();
      let batch = self.scheduler.build_batch();
      if batch.is_empty() {
          return Ok(vec![]);
      }

      // PLAN 17.4-E: Warmup draft KV cache after prefill
      if batch.phase == BatchPhase::Prefill && self.speculative_mode {
          self.warmup_draft_kv(&batch)?;
          tracing::debug!(
              seq_count = batch.seq_ids.len(),
              phase = ?batch.phase,
              "Draft KV cache warmed up"
          );
      }

      // ... rest of existing step_speculative_inner logic ...
  }
  ```

- The `warmup_draft_kv()` method should only run the draft forward for sequences that are in speculative mode. Add a check: `self.speculative_mode` is already checked above.

- Edge case: if there's no draft model configured (`self.draft_model.is_none()`), skip warmup entirely.

- Edge case: if the draft forward fails during warmup, log a warning and continue (non-fatal — the speculative path will fall back on next step):
  ```rust
  if let Err(e) = self.warmup_draft_kv(&batch) {
      tracing::warn!(error = %e, "Draft warmup failed, continuing without warmup");
  }
  ```

## Risk Assessment

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Warmup forward with full input is expensive for long prompts | Increased prefill latency | Medium | Only runs one extra draft forward per prefill step, which is cheap (1/8 layers);
| Draft KV block IDs may not match across target and draft forward passes | KV cache mismatch | Medium | Draft and target share same block IDs per sequence by design — the scheduler allocates blocks, both models write to those block IDs |
| Per-request acceptance tracking via DashMap seq_id keys may grow unbounded | Memory leak | Low | Remove entries when sequences finish (already done in finished_sequences cleanup) |
| Warmup failure should not crash the engine | Silent fallback to non-speculative | Low | Wrapped in `if let Err(e) = ...` with warning log; step continues without draft |
| Efficiency metric of 0 after first step (no draft tokens yet) | Confusing zero values | Low | Efficiency is 0 until at least one speculative step has draft tokens — this is correct behavior |
