# Phase 18.2: Lifecycle + Memory Budget - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning
**Mode:** Autonomous (smart-discuss)

<domain>
## Phase Boundary

Add lifecycle and memory-budget enforcement on top of the v18.1
`DraftModelRegistry`:

1. **Refcount-driven lifecycle (LIFE-02/03):** `unload` checks refcount; if a
   draft is in use, return `InUse(n)` and refuse. `decrement_ref` auto-unloads
   when refcount reaches zero. `unload` itself becomes "force unload" via an
   explicit opt-in flag.

2. **VRAM budget enforcement (MEM-01..03):** the Engine tracks a configured
   `vram_budget_bytes` (or `vram_budget_gb`). Before calling `attach_loaded`,
   the caller (Engine or scheduler) asks the registry "would loading this
   draft fit?". The registry estimates the draft's footprint from its
   `DraftSpec` (kv_blocks Ă— bytes_per_block + declared model size estimate)
   and rejects with a structured `MemoryBudgetExceeded` error if the new total
   would exceed the budget. Runtime KV-cache growth tracking is added to each
   loaded draft so the engine can refuse new block allocations when the draft
   would push past its share of the budget.

3. **Block-allocator lifecycle integration:** when a draft is unloaded, its
   private `BlockAllocator` is dropped â€” all its block ids become invalid in
   one step. No orphan-block path exists because the allocator owns the
   block-id space.

</domain>

<decisions>
## Implementation Decisions

### Refcount semantics
- `increment_ref` and `decrement_ref` already exist (Phase 18.1); we extend them
  to drive auto-unload
- `decrement_ref` auto-unloads when count reaches zero AND the draft is in
  `Loaded` state â€” `Unloaded` drafts are simply decremented to zero with no
  side effect (consistent with the current floor-at-zero behavior)
- `unload` (forced) is split into:
  - `unload(id)` â€” fails with `InUse(refcount)` if `refcount > 0`
  - `force_unload(id)` â€” unconditional; for tests and admin tooling
- A new helper `try_release(id)` decrements refcount and auto-unloads if zero;
  this is what Phase 18.3 routing will call when a request finishes

### Memory budget tracking
- New `MemoryBudget` struct: `{ total_bytes: u64, reserved_target_bytes: u64, reserved_drafts_bytes: u64 }`
- `MemoryBudget::try_reserve_draft(spec) -> Result<usize, MemoryBudgetExceeded>` â€” returns reserved bytes if it fits
- `MemoryBudget::release_draft(bytes)` â€” frees a previously reserved amount
- Each `DraftSpec` carries a `weight_size_estimate_bytes: u64` field populated by the caller (from ModelLoader metadata, per MEM-02)
- `DraftModelRegistry` holds an `Arc<MemoryBudget>` shared with the Engine

### Weight-size estimation (MEM-02)
- `DraftSpec` gains `weight_size_estimate_bytes: u64` (mandatory for new specs)
- Caller (server, after `ModelLoaderBuilder.build()`) calls
  `loader.detected_capabilities()` and reports `weight_size_bytes` before
  `attach_loaded`
- Estimation is conservative â€” over-estimate rather than under-estimate
- For the registry itself, we don't read disk; we trust the caller's number
- In tests, the caller passes an explicit estimate (e.g., 1 GiB for a fake model)

### KV cache growth tracking (MEM-02)
- Each `LoadedDraft` already owns a `BlockAllocator`; we extend with
  `BlockAllocator::current_allocated_bytes() -> usize` (sum of allocated block sizes Ă— block_size)
- The registry exposes `draft_allocated_bytes(id) -> Result<u64, _>` for the
  Engine to query during runtime KV growth decisions
- This is **read-only** tracking in 18.2 â€” the Engine uses the figure to
  decide whether new allocations are safe; we don't auto-evict or pre-empt

### Refused-load error type (MEM-03)
- New `MemoryBudgetExceeded { requested_bytes, available_bytes, draft_id: Option<DraftId> }`
- Exposed as `DraftRegistryError::MemoryBudgetExceeded(MemoryBudgetExceeded)`
- The caller can pattern-match on the variant and surface a structured error
  (e.g., HTTP 507 Insufficient Storage in the server)

### No external KV-block sharing
- Refuse-load is the only enforcement â€” we do NOT preempt other drafts to
  make room (consistent with the target's preemption policy in v17)
- v18.2 budget is "best-effort" upper bound; overruns are possible only via
  spec misconfiguration, not via runtime KV growth

### MemoryBudget is in `crates/core/src/speculative/memory_budget.rs`
- Stays in `speculative` module since it's tightly coupled to drafts
- No dependency on vllm-model (bytes-only, no tensor access)

### the agent's Discretion
- Whether `MemoryBudget` lives in a separate struct vs as a field on
  `DraftModelRegistry` (chosen: separate, easier to share Arc with Engine)
- Whether to expose `MemoryBudget` snapshots via `Debug` or a dedicated
  `snapshot()` method (chosen: `snapshot()` returning a public struct)
- Exact arithmetic for byte calculation per block

</decisions>

<specifics>
## Specific Ideas

- `BLOCK_SIZE_BYTES = 16 * 1024 * 1024` (16 MiB per KV block, matches the
  project's existing block sizing â€” see SchedulerConfig defaults)
- `MemoryBudget` default is `u64::MAX` (no limit) so existing callers don't
  break
- `MemoryBudget::try_reserve_draft` is idempotent â€” calling twice reserves
  twice; caller must pair with `release_draft` on unload
- `unload` failure when in-use is a recoverable error: caller logs and
  defers; force-unload is logged at WARN level

</specifics>

<deferred>
## Deferred Ideas

- Preemption of target/draft sequences to free KV for a new draft â€” out of
  scope; would need cross-pool eviction
- Per-draft KV cache hot-reload (resize allocator on the fly) â€” v19.0+
- Disk-spilling over-budget drafts â€” out of scope, this is in-memory only
- Multi-GPU VRAM tracking (only sees one device's budget) â€” v19.0+

</deferred>
