# Phase 17.1: Engine Integration - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Engine executes correct speculative decode with batched draft generation, logit-based verification, KV rollback, and graceful fallback. Phase 17.1 wires the existing v16.0 speculative decode architecture (DraftVerifier trait, SpeculativeModel, SelfSpeculativeModel, SpeculationConfig, RejectionStrategy) into the main Engine inference loop, replacing the current per-sequence stub implementations with a real batched speculative path.

Key deliverables:
- Unified `step(max_draft)` entry point routing between speculative and non-speculative paths
- Batched per-position draft generation (not per-sequence loop)
- Logit-based probability comparison for token verification (not exact match)
- KV cache rollback via MemoryManager for rejected draft tokens
- Graceful fallback to non-speculative decode on draft model error
- Scheduler correctly tracks input token counts when multiple draft tokens are accepted per step
</domain>

<decisions>
## Implementation Decisions

### Speculative Step Architecture
- Introduce a single `step(max_draft: Option<usize>)` method on Engine that internally routes to speculative or non-speculative paths ‚Äî avoids branching at the call site
- KV rollback belongs in `MemoryManager::rollback(seq_id, num_tokens)` ‚Äî the scheduler already owns memory management
- On draft model error: catch the error in `generate_draft_tokens`, log a warning, and fall through to non-speculative `step()` for the current iteration
- Do NOT add `forward_with_logits()` to ModelBackend ‚Äî use the existing `forward_logits()` for verification

### Batched Draft Generation
- Replace the current per-sequence loop with a batched per-position forward pass: all sequences generate draft position k before advancing to k+1
- Use `max_draft_tokens` globally across all sequences in the batch (not per-sequence configurable)
- Pad to max draft length ‚Äî short sequences produce fewer drafts but participate in every position batch

### Logit-Based Verification & Scheduler Tracking
- Use logit-based comparison: compare draft model probabilities vs target model probabilities via `forward_logits()` (rejection sampling style, not exact match)
- Pass actual accepted draft token count per sequence as `input_counts` to `scheduler.update()` ‚Äî the existing API already supports values > 1
- Wire `SelfSpeculativeModel` (from `speculative/self_spec.rs`) as the DraftVerifier in the engine's speculative path

### the agent's Discretion
Implementation details of the batched per-position draft loop, specific MemoryManager rollback API design, and error handling granularity are left to the agent's discretion.

</decisions>

<specifics>
## Specific Ideas

- The batched draft generation should work like: for position 0 across all seqs ‚Üí run draft forward ‚Üí collect tokens ‚Üí for position 1 across all seqs ‚Üí ... ‚Üí repeat up to max_draft
- Logit verification needs to call target_model.forward_logits() on the concatenated [input + drafts] to get logits, then compare draft_token vs target_token using the DraftVerifier.verify() trait method
- KV rollback should free the KV blocks allocated for rejected draft positions ‚Äî the scheduler already has `free_kv_blocks()` or similar
- The `step()` method signature `step(max_draft: Option<usize>)` means: None ‚Üí non-speculative, Some(0) ‚Üí also non-speculative, Some(n) ‚Üí speculative with up to n draft tokens
</specifics>
