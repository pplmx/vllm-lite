# Phase 18.3: Request Routing + Fallback - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning
**Mode:** Autonomous (smart-discuss)

<domain>
## Phase Boundary

Wire `Request` to a draft model and let the scheduler route each request to
the right draft at batch composition time, with graceful fallback semantics
when drafts fail.

1. **Per-request draft selection (RTE-01):** `Request` gains an optional
   `draft_model_id: Option<DraftId>` field. `SamplingParams` does NOT gain
   the field — placing it on `Request` is simpler and matches how the engine
   already threads `Request` through scheduling.

2. **Mixed-draft routing (RTE-02/03):** the Engine step loop, when in
   speculative mode, looks up the draft for each request and runs the
   speculative path with that draft. Multiple requests with different drafts
   coexist in the same batch — the scheduler groups by `draft_model_id` so
   the batch forward pass is per-draft, but the engine doesn't require
   exclusive drafts.

3. **Load-failure fallback (FALL-01):** if the requested draft fails to
   load (e.g., the registry returns `UnknownDraftId`, `MemoryBudgetExceeded`,
   or model loader fails), the engine silently falls back to the self-spec
   path (v17 baseline). The user is not aware unless they read metrics.

4. **Runtime error degradation (FALL-02):** if the draft model errors during
   inference (e.g., forward pass panics or returns an error), the affected
   request gracefully degrades to non-speculative decode for that step.
   Other requests in the batch are unaffected.

</domain>

<decisions>
## Implementation Decisions

### Request carries `draft_model_id`
- Add `pub draft_model_id: Option<DraftId>` to `Request`
- Default is `None` → uses self-spec or non-spec path, depending on engine mode
- `Request::new` leaves it as `None` for backward compatibility
- Add `Request::with_draft_model(self, id)` builder

### Resolution semantics
- `Request::draft_model_id == None` → no external draft (existing behavior)
- `Request::draft_model_id == Some(id)` and registry has the id loaded →
  speculative with that draft
- `Request::draft_model_id == Some(id)` and registry has the id unloaded →
  attempt `attach_loaded_budgeted` with caller-supplied loader (FALL-01 path)
- `Request::draft_model_id == Some(id)` and registry doesn't have the id →
  log + fall back to self-spec

### Step loop integration
- The Engine `step_speculative` already exists from v17 — extend it to look
  up the draft per-request instead of using the single `draft_model` field
- For each request in the batch:
  1. Resolve draft via `engine.resolve_draft(req)?` (returns `ResolvedDraft`)
  2. If `ResolvedDraft::External(backend)`, use it for that request's
     speculative draft generation
  3. If `ResolvedDraft::SelfSpec`, use the self-spec layer
  4. If `ResolvedDraft::None`, skip speculative decode for that request
- Multiple drafts in one batch are supported because each request carries
  its own draft selection

### `ResolvedDraft` enum
```rust
pub enum ResolvedDraft {
    External(Arc<Mutex<Box<dyn ModelBackend>>>), // multi-draft path
    SelfSpec(Arc<Mutex<Box<dyn ModelBackend>>>), // v17 baseline
    None, // no spec; pure target decode
}
```

### Load failure handling (FALL-01)
- When `resolve_draft` finds an unloaded id and `attach_loaded_budgeted`
  fails, the engine logs at WARN with the draft id + error, increments a
  `draft_load_failures_total` metric, and returns `ResolvedDraft::SelfSpec`
- The user sees the same throughput as the v17 self-spec path — no error
  surfaced to the client

### Runtime error handling (FALL-02)
- The speculative forward pass wraps each per-request draft call in a
  catch_unwind or `Result`-based error path
- On draft error, the affected request is processed with non-spec decode
  for this step (and subsequent steps, until the request completes)
- The engine increments `draft_runtime_errors_total` metric
- Other requests in the batch proceed normally

### Scheduler changes
- The scheduler doesn't need a new field — batch composition already groups
  sequences. We add a `BatchGroup::draft_model_id` field so the engine
  knows which draft to use per group
- `BatchComposer` groups by `(draft_model_id, status)` so requests with
  different drafts form different sub-batches

### Metrics
- New counters:
  - `draft_resolutions_total { result: "external" | "self_spec" | "none" }`
  - `draft_load_failures_total`
  - `draft_runtime_errors_total`
- These land in the existing `EnhancedMetricsCollector` (no new metrics
  framework needed)

### Error type
- New `DraftResolutionError` enum: `UnknownDraftId`, `LoadFailed(String)`,
  `BudgetExceeded` — but these are NEVER surfaced to callers; the resolver
  always degrades to a safe `ResolvedDraft`
- Surfaced only via `tracing::warn!` + metrics counters

### the agent's Discretion
- Exact metric counter names (chosen: snake_case, consistent with existing
  metrics)
- Whether to expose `ResolvedDraft` publicly or hide it as an internal type
  (chosen: internal, behind `pub(crate)`)
- Whether `Request::with_draft_model` accepts `impl Into<DraftId>` (chosen: yes, ergonomics)

</decisions>
