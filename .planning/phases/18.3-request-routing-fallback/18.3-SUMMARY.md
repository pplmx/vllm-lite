- `ResolvedDraft::{External, SelfSpec, None}` — the three possible resolution outcomes
- `DraftLoader` trait — abstracts the actual `ModelLoader` call so the resolver has no `vllm-model` dependency
- `DraftResolver::resolve(request_draft_id)` — implements FALL-01 (load failure → SelfSpec) and the routing logic

**Resolution algorithm:**
1. `request_draft_id == None` → `SelfSpec` (if available) or `None`
2. Draft already loaded → `External(backend_arc)`
3. Draft registered but unloaded → invoke loader → attach via `attach_loaded_budgeted` → `External`
4. Loader fails / budget exceeded / unknown id → log + metric + `SelfSpec` fallback

## Design Notes

1. **Per-request resolution, not per-batch** — the resolver doesn't pre-compute
   a "batch's draft". Each request in a batch resolves independently, so mixed
   routing (RTE-03) is automatic. The step loop will iterate requests and
   resolve each one (the integration hook in `Engine::step_speculative` is the
   next place to wire this in — left as a follow-up since the resolver itself
   is testable in isolation).

2. **DraftLoader is a trait, not a closure** — the resolver accepts
   `Arc<dyn DraftLoader>` so the server can pass a closure-based loader
   without `vllm-core` gaining a `vllm-model` dependency. The trait surface
   is tiny (`fn load(&DraftId) -> Result<Box<dyn ModelBackend>, _>`).

3. **Sequence carries the draft_model_id** — propagating from `Request` to
   `Sequence` at enqueue time means the resolver can be called from any code
   path that has access to a `Sequence`, not just the engine step loop. This
   makes the routing logic composable.

4. **FALL-02 is not yet integrated into the speculative step loop** — the
   resolver handles FALL-01 (load-time), but runtime draft errors during the
   speculative forward pass need a wrapping helper at the engine step level
   that catches `Err` from `backend.forward()` and sets `degraded_draft = true`
   on the affected sequence. This is a follow-up: Phase 18.4 integration tests
   will exercise this path end-to-end.

5. **Metrics counter design** — three separate counters for the three
   resolution outcomes (external / self_spec / none) so dashboards can show
   fallback rate as `self_spec / (external + self_spec + none)` and load
   failure rate as `load_failures_total / (external + self_spec + none)`.
   Runtime errors are tracked separately because they indicate a different
   failure mode than load failures.
