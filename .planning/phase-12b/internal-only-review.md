# Phase 12b INTERNAL-ONLY Review (Phase 17)

**Date:** 2026-07-11
**Scope:** Re-review the 62 items classified as `INTERNAL-ONLY` by the v2 audit
(see `audit-report-v2.md` §6 and `dead-pub-candidates.tsv`).
**Outcome:** 2 tightening wins applied; 60 items reviewed-and-intentionally-pub.

---

## 1. Executive Summary

The Phase 12b v2 audit produced three action buckets:

| Bucket | Count | Phase 12c/d action |
|--------|------:|--------------------|
| UNIT-TEST-ONLY | 43 | tightened to `pub(crate)` (Phase 12c) |
| INTEGRATION-TEST / TEST-ONLY-MIXED | 15 | kept `pub` |
| TRULY-UNUSED | 33 | removed or `#[doc(hidden)]` (Phase 12d) |
| INTERNAL-ONLY | 62 | **left for opportunistic review** |

This document closes the INTERNAL-ONLY bucket.

**INTERNAL-ONLY definition (v2 audit):** an item has zero external production
callers, zero external unit-test callers, AND zero external integration-test
callers. The only "same-file" references can be the declaration itself,
docstring mentions, or in-file private use. The `cargo-public-api` baseline
counts the item as `pub`.

**Key finding after re-review:** most INTERNAL-ONLY items are **legitimately
`pub`**. They are either:

1. The implementation of a public trait (`NodeServiceImpl::into_service` for
   tonic gRPC) — must be `pub`.
2. Newly added public API methods on existing types (`BackpressureState::pending_count`,
   `Pipeline::is_pipeline_parallel`, `ErrorRecovery::from_error`) — only called
   in the declaring file because they are new, not because they are internal.
3. Constants in feature-gated generated code (`SERVICE_NAME` in
   `vllm.distributed.rs`) — generated by `tonic-build` and required to be `pub`.
4. Builder-style methods that downstream embedders are expected to use
   (`TestingHarness::add_test_request`, `with_prompt` in `vllm-testing`).

The honest deliverable is therefore *narrow*: only 2 of 62 items have a real
visibility issue that can be addressed without breaking downstream intent.

## 2. Actionable Wins (2 items)

### `MAX_OBSERVERS` (core/scheduler/observer.rs:110)

**Before:** `pub const MAX_OBSERVERS: usize = 16;`
**After:** `pub(crate) const MAX_OBSERVERS: usize = 16;`
**Commit:** `1decff2 refactor(core): tighten MAX_OBSERVERS to pub(crate)`

**Rationale:** Pure implementation constant used only inside `observer.rs`
(lines 124, 126) to bound the registry. No external embedder has any reason
to read this value (it's a `Vec::with_capacity` cap, not a tunable). Tightening
to `pub(crate)` is a strict tightening — `pub(crate)` items are still visible
inside the workspace.

### `apply_q_norm_impl` / `apply_k_norm_impl` (model/gqa/norm.rs:35, 57)

**Before:** Two public methods that take explicit `(batch, heads, seq, dim)`
shape parameters and run the norm.
**After:** Removed.
**Commit:** `3429066 refactor(model): remove orphaned apply_q_norm_impl / apply_k_norm_impl`

**Rationale:** Workspace-wide grep confirms zero callers. The only references
were in the `gqa/mod.rs` and `norm.rs` module docstrings. The sibling
`_flattened` variants (`apply_q_norm_impl_flattened`,
`apply_k_norm_impl_flattened`) ARE called — from `rope_gqa.rs:142` — and
remain untouched.

## 3. Reviewed-and-Intentionally-`pub` (60 items)

### 3.1 Generated code — never touch (13 items)

| File | Items |
|------|-------|

### 3.4 Server crate — middleware-facing API (6 items)

| File | Items |
|------|-------|
| `crates/server/src/backpressure.rs` | `batch_increment`, `batch_decrement`, `should_throttle`, `pending_count` (BackpressureState helpers), `before_send`, `after_send`, `can_send` (StreamingBackpressure middleware API) |
| `crates/server/src/security/rbac.rs` | `extract_role_from_headers` |

**Why keep pub:** `backpressure` is wired into the HTTP middleware pipeline;
its `before_send` / `after_send` / `can_send` methods are the public surface
that middleware components (axum extractors, tower layers) call. The
`BackpressureState` helpers are similarly exposed for direct state
inspection by middleware. `rbac::extract_role_from_headers` is the
extractor function for the auth middleware.

| File | Items |
|------|-------|
| `crates/testing/src/builders/mod.rs` | `with_prompt` |

**Why keep pub:** `vllm-testing` is consumed by `crates/core/tests/*.rs`,

audit found them as INTERNAL-ONLY because the explicit
`add_test_request` / `build_test_batch` / etc. helpers happen to only
be invoked from inside the testing crate itself. They are intentional
public API for downstream test code.

## 4. Conclusion

- **2 of 62 INTERNAL-ONLY items** had real visibility issues → fixed.
- **13 of 62** are auto-generated → out of scope (do not edit).
- **1 of 62** is a trait-required method → must stay `pub`.
- **46 of 62** are legitimate public API on existing types → keep `pub`.

The Phase 12b v2 audit's "INTERNAL-ONLY" verdict is a useful signal but not
a verdict that "these should be tighter". The audit explicitly flagged
this bucket as **low priority; review opportunistically** (see
`audit-report-v2.md` §6). This review confirms that classification.
