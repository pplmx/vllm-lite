# Phase 25-02: Feature-gate `vllm-core → vllm-model` (cuda-graph) — ARCH-F-12

## Objective

Confirm and harden the existing feature gate that makes the `vllm-core → vllm-model` dependency edge (ARCH-F-12) optional, so `cargo build -p vllm-core --no-default-features` does not require `vllm-model` at all, and only `--features cuda-graph` activates the edge.

## Files to Modify

| File | Lines | Change |
|------|------:|--------|
| `crates/core/Cargo.toml` | 25-29 | Verify `[features].cuda-graph = ["dep:vllm-model"]` + optional `[dependencies.vllm-model]` block (already done) |
| `crates/core/src/engine.rs` | 27, 30 | Verify `#[cfg(feature = "cuda-graph")] use vllm_model::kernels::BatchCudaGraphExecutor;` and similar (already done) |
| `crates/server/Cargo.toml` | 20 | Verify `vllm-core = { path = "../core", features = ["cuda-graph"] }` is **only** activated when server needs CUDA-graph executor (confirm the server build does not inadvertently enable `cuda-graph` for everyone) |

**Audit references:** `.planning/audit/architecture/REPORT.md` §4.3.2 (ARCH-F-12); `crates/core/Cargo.toml:25-29`; `crates/core/src/engine.rs:27`.

## Tasks

### Task 1: Audit current state of the cuda-graph feature gate

Verify the three pieces of the gate are in place:

```bash
# (a) Cargo.toml: vllm-model is optional, gated by cuda-graph
grep -A2 "vllm-model" crates/core/Cargo.toml
# Expected:
#   [dependencies.vllm-model]
#   path = "../model"
#   optional = true
# And: cuda-graph = ["dep:vllm-model"]

# (b) engine.rs: all vllm_model imports are cfg-gated
grep -n "vllm_model" crates/core/src/engine.rs
# Expected: every match is preceded by #[cfg(feature = "cuda-graph")]

# (c) server's cuda-graph activation: is it on by default?
grep "cuda-graph" crates/server/Cargo.toml
# Expected: NOT in default features; only enabled conditionally (e.g., behind a server feature)
```

If any of the above is not satisfied, fix before proceeding.

### Task 2: Verify default-feature build excludes vllm-model

```bash
# Build vllm-core with default features; vllm-model must NOT appear in its tree
cargo tree -p vllm-core --no-default-features | grep -c vllm-model
# Expected: 0

# Build vllm-core WITHOUT default features; vllm-model must NOT appear
cargo tree -p vllm-core --no-default-features --features 'prometheus' | grep -c vllm-model
# Expected: 0 (default features = ["prometheus"], still excludes vllm-model)

# Build vllm-core WITH cuda-graph; vllm-model MUST appear
cargo tree -p vllm-core --features cuda-graph | grep -c vllm-model
# Expected: ≥1
```

**Acceptance:** `0` for default builds, `≥1` with `--features cuda-graph`.

### Task 3: Confirm server does not unconditionally pull cuda-graph

The server is the only consumer that ever enables `cuda-graph`. If server enables it by default, the gate is effective only for `cargo build -p vllm-core` in isolation — the layering violation persists in the actual binary.

```bash
# Inspect server's feature resolution
cargo tree -p vllm-server | grep -c vllm-model
# Expected: ≥1 (server depends on vllm-model directly)

# Confirm vllm-core is built with cuda-graph by server
cargo tree -p vllm-server -e features | grep -B1 -A1 vllm-model | head -20
# Expected: show that vllm-model is reached via vllm-core's cuda-graph feature OR via direct server dep
```

Document the result: if server enables `cuda-graph` by default, this plan should propose moving that activation behind a server feature flag. If not, ARCH-F-12 is fully resolved and the plan is a no-op verification.

### Task 4: Confirm default-feature build is healthy

```bash
cargo build -p vllm-core
cargo build -p vllm-core --no-default-features
cargo build -p vllm-core --features cuda-graph
# All three must exit 0
```
